# Profiling findings — full matrix, 2026-06-13

Re-profiled all 6 variants on all 3 platforms against the current binaries
(JVM via JFR @250 steps, JS via `node --cpu-prof` @50 steps with a 12 GB heap,
Native via PerfView/ETW @40 steps). Collector (Jaeger + Envoy) up throughout.

Headline: the picture has changed since the older profiling notes. The old
"`isValidBase16String` ≈ 56% of JVM self-time" is no longer true. Span-context
validation is now a few percent. The dominant cost is a **quadratic `removeAll`
in the BatchSpanProcessor's batch list, and it shows up on both the JVM and JS**
— it is an algorithmic bug in the shared Kotlin SDK code, not a platform quirk.

## The cross-platform bug

The `BatchSpanProcessor` stores its in-flight batch as a `kotlinx.collections.immutable`
persistent vector. After each export it calls `removeAll` to drop the exported
spans, and `removeAll` on a persistent vector does a linear `contains` / `indexOf`
scan per element — O(batch²). With a 2048-span batch that's ~4M element
comparisons per flush.

- **JVM otel-proto:** `Intrinsics.areEqual` ≈ 48% of CPU, reached entirely via
  `PersistentVectorBuilder.recyclableRemoveAll → AbstractPersistentList.contains
  → AbstractList.indexOf`.
- **JVM otel-proto-sampler:** same shape, `areEqual` ≈ 51%.
- **JS otel-proto:** `recyclableRemoveAll` is the #2 self-cost; `removeAll` is
  the largest single *inclusive* cost (~7.6 s of a 45 s run).
- It also runs on the application thread: the BSP worker uses an unconfined
  coroutine dispatcher (`DispatchedTaskKt.resumeUnconfined → BatchSpanProcessor$Worker.run`
  in the JVM stacks), so the cleanup resumes on the thread that called `span.end()`.

Fix: use a plain `ArrayList` (or `clear()` the batch after export) instead of a
persistent vector + `removeAll`. O(B²) → O(B). This is an upstream fix to
dcxp/opentelemetry-kotlin that would help every platform at once, and it's the
single biggest lever found.

## Per-platform / per-variant hot paths

| Variant | Platform | Dominant cost |
|---|---|---|
| k-perf | JVM | `Utf8Kt.writeString` (the buffered trace-file write) + number formatting. The instrumentation IS the file write, as advertised. |
| otel-proto / sampler / timesource / anchored | JVM | quadratic `removeAll` (~50%), then protobuf `Span.<init>` (~12%); validation ~3-4%; clocks negligible. |
| otel (JSON) | JVM | JSON text encoding: `UTF_8$Encoder.encodeBufferLoop` ~25% + `JsonToStringWriter.writeQuoted` ~14%; `removeAll` still ~16%. |
| otel-proto | JS | three co-equal taxes: the `removeAll`, 64-bit `Long`-polyfill arithmetic (`divide`/`subtract`/`multiply`/`bitwiseAnd`/`equalsLong`/`lessThan` — JS has no native i64, span timestamps are i64 nanos), and heavy GC from span retention. The ktor client is also significant. |
| otel (JSON) | JS | ktor HTTP client (`ktor-client-core` ~27%) + GC (~18%) + JSON/UTF-8 encoding + `removeAll` + Long math. Slowest variant overall. |
| all | Native | uninformative at function level: Kotlin/Native release builds strip symbols, so ~87% of time is one opaque `main!?` blob. Confirms the cost is in-process compute (network threads are tiny), but cannot name functions. |

## What this explains

- **Why protobuf beats JSON on the JVM (~29%):** JSON pays a large text-encoding
  tax (UTF-8 encode + quote/escape) that the binary protobuf format skips. Clean
  mechanism for the headline result.
- **Why the sampler only saved ~6%:** the validation it removes is ~3-4% of cost;
  the dominant `removeAll` is downstream of sampling and untouched. ~6% is the ceiling.
- **Why timestamp variants gained nothing:** clocks are negligible in the profile
  (~11 samples on JVM). The cost was never the clock.
- **Why the transport win disappears on JS:** the JS bottleneck is the `removeAll`,
  64-bit Long polyfill, and GC — none of which the wire format changes.
- **Why delivering spans costs ~7× vs dropping them:** dropped spans never fill or
  drain the batch, so the quadratic `removeAll` (the bulk of the on-thread cost)
  never runs.

## Caveats

- Native otel-proto-sampler and otel-proto-anchored failed to capture (PerfView
  exit 1 in the elevated batch; likely a collision because all otel native
  binaries share the process name `main.exe`). Not re-run: native is uninformative
  anyway due to stripped symbols.
- k-perf JVM produced only ~20 JFR samples (it runs in ~0.2 s at 250 steps);
  enough to confirm the file-write hot path, not for fine attribution.
- Step counts differ per platform (JVM 250 / JS 50 / Native 40) to keep JS off
  the V8 heap ceiling and native runs short; absolute times are not comparable
  across platforms here, only the relative frame breakdown within each profile.
